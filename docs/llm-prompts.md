# AI需求分析提示词文档

## 一、需求拆解提示词

### 1. 核心提示词模板

```json
{
  "name": "AI需求分析 - 需求拆解",
  "description": "将原始需求拆解为具体功能点并生成追问问题",
  "category": "ai-requirement-analysis",
  "template": "你是一个资深产品经理，请分析以下原始需求，将其拆解为具体的功能点，并生成需要追问的问题：\n\n原始需求：{{requirementText}}\n\n请按照以下JSON格式输出：\n{\n  \"analysisResults\": [\"需求点1\", \"需求点2\", \"需求点3\", \"需求点4\"],\n  \"questions\": [\"追问问题1\", \"追问问题2\"]\n}\n\n分析要求：\n1. 识别需求中的核心功能点\n2. 识别性能、安全等非功能需求\n3. 识别模糊词汇，生成追问问题\n4. 每个需求点应该是具体可执行的任务\n5. 追问问题应该帮助澄清模糊点",
  "variables": ["requirementText"],
  "isDefault": true,
  "isActive": true
}
```

### 2. 算法流程

1. **输入接收**：接收原始需求文本
2. **模板渲染**：将原始需求文本填充到提示词模板中
3. **LLM调用**：将渲染后的提示词发送给大语言模型进行分析
4. **结果解析**：解析LLM返回的JSON结果
5. **数据提取**：提取拆解后的需求点和追问问题
6. **结果返回**：返回结构化结果供人工确认

### 3. 示例

**输入**：
```
销售团队反映手机登录太慢，网络不好时经常超时
```

**输出**：
```json
{
  "analysisResults": [
    "登录接口响应时间<2秒 (性能)",
    "弱网超时重试机制 (功能)",
    "网络异常友好提示 (UI/UX)"
  ],
  "questions": [
    "登录速度具体要求是多少秒？",
    "弱网环境的定义是什么？",
    "重试机制的具体参数（次数、间隔）是什么？"
  ]
}
```

## 二、用户故事生成提示词

### 1. 核心提示词模板

```json
{
  "name": "需求转用户故事 - 默认模板",
  "description": "将需求转换为标准用户故事格式",
  "category": "requirement-to-story",
  "template": "你是一个专业的软件需求分析师。请将以下需求转换为标准用户故事格式。\n\n请严格遵循以下格式：\n\"作为[角色]，我想要[功能]，以便[价值]\"\n\n需求：{{requirement}}\n\n请确保：\n1. 角色明确具体\n2. 功能清晰可执行\n3. 价值体现业务目标\n\n请输出JSON格式：\n{\n  \"userStories\": [\n    {\n      \"role\": \"具体角色\",\n      \"feature\": \"具体功能\",\n      \"value\": \"业务价值\"\n    }\n  ]\n}",
  "variables": ["requirement"],
  "isDefault": true,
  "isActive": true
}
```

### 2. 示例

**输入**：
```
登录接口性能优化，要求95%请求响应时间 < 2秒，99%请求响应时间 < 3秒
```

**输出**：
```json
{
  "userStories": [
    {
      "role": "移动端销售员",
      "feature": "在2秒内完成登录",
      "value": "快速查看客户信息，不错过商机"
    }
  ]
}
```

## 三、验收标准生成提示词

### 1. 核心提示词模板

```json
{
  "name": "AI需求分析 - 生成验收条件",
  "description": "生成Given-When-Then格式的验收条件",
  "category": "ai-generate-acceptance-criteria",
  "template": "你是一个资深测试工程师，请根据以下需求内容，生成3-5个Given-When-Then格式的验收条件：\n\n需求内容：{{requirementContent}}\n\n请输出JSON格式，包含acceptanceCriteria数组，每个验收条件包含given、when、then和scenarioType字段。",
  "variables": ["requirementContent"],
  "isDefault": true,
  "isActive": true
}
```

### 2. 示例

**输入**：
```
登录接口性能优化，要求95%请求响应时间 < 2秒，99%请求响应时间 < 3秒
```

**输出**：
```json
{
  "acceptanceCriteria": [
    {
      "given": "系统处于正常负载状态",
      "when": "用户发起登录请求",
      "then": "95%的请求响应时间应小于2秒",
      "scenarioType": "normal"
    },
    {
      "given": "系统处于高负载状态（接近200并发）",
      "when": "大量用户同时发起登录请求",
      "then": "99%的请求响应时间应小于3秒",
      "scenarioType": "boundary"
    }
  ]
}
```

## 五、提示词优化策略

### 1. 模型适配
- 针对不同LLM模型（DeepSeek、OpenAI、Ollama等）优化提示词语言风格
- 调整输出格式要求，确保各模型能正确理解和生成JSON格式

### 2. 性能优化
- 保持提示词简洁明了，避免冗余内容
- 使用明确的格式要求，减少解析错误

### 3. 效果提升
- 加入领域特定知识，提高分析准确性
- 持续收集反馈，迭代优化提示词内容

## 六、使用说明

1. **需求拆解**：在原始需求录入后自动调用，生成初步需求点和追问问题
2. **用户故事生成**：在需求定义完成后，基于需求规格生成用户故事
3. **验收标准生成**：与用户故事生成同时进行，确保两者一致性
4. **质量评分**：在需求提交评审前进行，确保需求质量符合标准
