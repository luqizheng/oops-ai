# 需求生命周期

## 一、正确的需求生命周期顺序

```text
时间轴 → → → → → → → → → → → → → → → → → → → → → → → → → → → → → → →

阶段1: 原始需求录入
    ↓
阶段2: 需求分析 (AI拆解 + 人工确认)
    ↓
阶段3: 需求定义 (PRD/规格说明书)
    ↓
阶段4: 故事编写
    ↓
阶段5: 需求评审
    ↓
后续阶段...
```

核心原则：必须人工确认分析结果后，才能进入需求定义阶段。

## 二、正确的阶段划分和输出

| 阶段                | 核心活动          | 输入       | 输出           | 是否需人工确认  |
| ------------------- | ----------------- | ---------- | -------------- | --------------- |
| 阶段1: 原始需求录入 | 记录客户原话      | 客户沟通   | 原始需求记录   | ✅ 确认内容准确 |
| ↓                   | ↓                 | ↓          | ↓              | ↓               |
| 阶段2: 需求分析     | AI拆解 + 人工调整 | 原始需求   | 结构化需求列表 | ✅ 必须确认     |
| ↓                   | ↓                 | ↓          | ↓              | ↓               |
| 阶段3: 需求定义     | 撰写详细规格      | 结构化需求 | PRD/需求文档   | ✅ 确认完整性   |
| ↓                   | ↓                 | ↓          | ↓              | ↓               |
| 阶段4: 故事编写     | 用户视角描述      | 需求文档   | 用户故事       | ✅ 确认价值     |
| ↓                   | ↓                 | ↓          | ↓              | ↓               |
| 阶段5: 需求评审     | 团队评审          | 用户故事   | 评审纪要       | ✅ 团队确认     |

## 三、每个阶段的详细说明

### 阶段1: 原始需求录入

```yaml
目标: 完整记录客户原始表述
负责人: 产品经理/业务方
输出: 原始需求记录

示例:
📝 原始需求内容:
"销售团队反映手机登录太慢，网络不好时经常超时"

📎 元数据: - 提出人: 销售总监 - 时间: 2024-03-15 - 场景: 周会反馈 - 附件: 会议录音

动作:

- 保存原始内容 (不可修改)
- 触发AI分析

```

### 阶段2: 需求分析 (关键的人工确认点)

```yaml
目标: 将原始需求拆解为清晰、独立的需求点
负责人: 产品经理 + AI辅助
输入: 原始需求
输出: 结构化需求列表

流程:

1. AI自动拆解原始需求
2. 生成初步的结构化需求列表
3. **人工审核和调整** ← 关键确认点
4. 确认最终的需求列表

示例:
原始需求: "登录太慢，网络不好时超时"

AI拆解结果:
├── 需求A: 登录接口响应时间<2秒 (性能)
├── 需求B: 弱网超时重试机制 (功能)
└── 需求C: 网络异常友好提示 (UI/UX)

人工调整:
├── ✅ 需求A 保留 (确认指标)
├── ✅ 需求B 保留 (补充重试次数)
├── ❌ 需求C 合并到需求B (过于细化)
└── ➕ 新增需求D: 登录失败日志记录 (安全)

确认后输出:
[
{
"id": "REQ-001",
"title": "登录接口性能优化",
"type": "性能需求",
"priority": "高"
},
{
"id": "REQ-002",
"title": "弱网环境优化",
"type": "功能需求",
"priority": "中"
}
]
```

### 阶段3: 需求定义 (基于已确认的分析结果)

```yaml
目标: 将每个需求写成完整的规格说明
负责人: 产品经理
输入: 已确认的结构化需求列表
输出: PRD/需求规格说明书

前提: **必须先确认阶段2的分析结果**

示例:
基于 REQ-001 撰写详细规格:

# 需求规格: 登录接口性能优化 (REQ-001)

## 详细描述

优化登录接口响应时间，提升用户体验

## 验收标准

- 95%请求响应时间 < 2秒
- 99%请求响应时间 < 3秒
- 支持200并发

## 业务规则

- 超时3秒自动重试1次
- 记录超时日志

## 依赖
- 需要网关支持超时配置
```

### 阶段4: 故事编写

```yaml
  目标: 从用户视角描述需求价值
  负责人: 产品经理 + 团队
  输入: 需求规格
  输出: 用户故事

示例:
基于 REQ-001 编写用户故事:

📖 作为 移动端销售员
📖 我想要 在2秒内完成登录
📖 以便 快速查看客户信息，不错过商机

✅ 验收标准:

- 4G网络下登录 < 2秒
- 弱网时有加载动画
- 失败时有明确提示
```

## 四、状态流转图（修正版）

```text
原始需求录入 → 待分析 → AI分析完成 → 人工确认中 → 已确认 → 需求定义中 → 已定义 → 故事编写中 → 已编写 → 待评审
  ↓            ↓           ↓            ↓          ↓         ↓            ↓        ↓            ↓        ↓
保存完整   触发AI分析  生成初稿     人工调整   分析完成  撰写规格    文档完成  编写故事   故事完成  准备评审

                                   ↑
                               **关键确认点**
                            必须人工确认后才能继续
```

## 五、在界面中的体现

正确的步骤顺序

```txt
步骤1: 原始需求录入  →  步骤2: AI需求分析  →  步骤3: 人工确认  →  步骤4: 需求定义  →  步骤5: 用户故事
                           ↓                     ↓                  ↓                 ↓
                        AI拆解需求           确认/调整         撰写详细规格        编写故事
```

## 六、总结：正确的阶段顺序

| 顺序 | 阶段         | 核心动作          | 关键点               |
| ---- | ------------ | ----------------- | -------------------- |
| 1    | 原始需求录入 | 记录客户原话      | 保持原样，不加工     |
| 2    | 需求分析     | AI拆解 + 人工确认 | 必须确认后才能继续   |
| 3    | 需求定义     | 撰写详细规格      | 基于已确认的分析结果 |
| 4    | 故事编写     | 用户视角描述      | 强调价值             |
| 5    | 需求评审     | 团队确认          | 达成共识             |
