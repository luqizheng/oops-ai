## 项目

1. 项目
   - 定义：一个项目是一个独立的软件产品或服务，由多个团队成员合作开发。
   - 示例：“一个基于 Web 的客户关系管理系统”。
   - 关系：一个项目可以有多个需求，每个需求都是对项目的功能或服务的描述。
2. 项目成员
   - 定义：项目成员是参与项目开发的人员，包括项目负责人、开发人员、测试人员等。
   - 示例：“项目负责人：张三 开发人员：李四、王五 测试人员：赵六”。
   - 关系：一个项目可以有多个项目成员，每个项目成员都有不同的角色和职责。
3. 项目角色
   - 定义：项目角色是项目成员在项目中的不同责任和权限。
   - 示例：“项目负责人：负责项目的管理和协调，包括需求分析、项目计划、资源分配等。开发人员：负责系统的编码和功能实现。测试人员：负责系统的测试和 bug 修复。”
   - 关系：一个项目成员可以有多个项目角色，每个项目角色都有不同的权限和责任。

## 需求相关
  
  需求都与项目相关， 一个项目可以有多个需求，每个需求都是对项目的功能或服务的描述。

1. 原始需求（Raw Requirement）
   - 定义：用户直接提出的需求，未经过分析或处理。
   - 示例：“用户需要能够登录系统”。
   - 关系：一个原始需求可以有多个需求，每个需求都是对原始需求的分析和处理。
2. 需求（Requirement）
   - 定义：经过分析和处理的需求，用于系统功能的实现。
   - 示例：“系统需要支持用户登录功能”。
3. 用户故事（User Story）
   - 定义：从用户的角度描述系统功能的一种方式，通常包含三部分：As a [type of user], I want [a feature] so that [a benefit].
   - 示例：“作为一个用户，我想要能够登录系统，以便我可以访问我的账户。”
   - 关系：一个需求可以有多个用户故事，每个用户故事都是对需求的具体描述。   
4. 用例（Use Case）
   - 定义：从用户的角度描述系统功能的一种方式，通常包含四部分：Actors（参与者）、Preconditions（前置条件）、Main Success Scenario（主要成功场景）、Alternative Scenarios（备选场景）。
   - 示例：“参与者：用户 前置条件：用户未登录 主要成功场景：用户输入用户名和密码，系统验证通过，用户登录成功 备选场景：用户输入错误的用户名或密码，系统提示错误信息”。
   - 关系：一个用户故事可以有多个用例，每个用例都是对用户故事的具体实现。
5. 验收标准（Acceptance Criteria）
   - 定义：用于验证系统是否满足需求的标准，通常包含多个子项。
   - 示例：“用户登录后，系统显示用户个人信息页面。”
   - 关系：一个用例可以有多个验收标准，每个验收标准都是对用例的具体验证。
6. 测试用例（Test Case）
   - 定义：用于验证系统是否满足验收标准的具体测试用例。
   - 示例：“测试用例1：用户输入正确的用户名和密码，系统验证通过，用户登录成功。”
   - 关系：一个验收标准可以有多个测试用例，每个测试用例都是对验收标准的具体验证。
7. 业务规则（Business Rule）
   - 定义：用于描述系统行为的规则，通常包含多个子项。
   - 示例：“用户只能在系统登录后才能访问个人信息页面。”
   - 关系：一个需求可以有多个业务规则，每个业务规则都是对需求的具体实现。
8. 非功能需求（Non-Functional Requirement）
   - 定义：用于描述系统性能、安全、可维护性等方面的需求。
   - 示例：“系统响应时间不能超过5秒。”
   - 关系：一个需求可以有多个非功能需求，每个非功能需求都是对需求的具体实现。
9. 依赖关系（Dependency）
   - 定义：用于描述系统中不同组件之间的依赖关系。
   - 示例：“系统登录功能依赖于用户数据库。”
   - 关系：一个需求可以有多个依赖关系，每个依赖关系都是对需求的具体实现。
